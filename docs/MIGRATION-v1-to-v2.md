# Migrating from spec-v1 to spec-v2

## Why migrate

spec-v2 cleanly separates **what you want** (`spec.json`) from **how you plan to build it** (`plan.json`). Requirements, decisions, and verification scenarios stay stable in `spec.json`; task breakdowns are a disposable byproduct of `/execute` that can be regenerated without polluting the spec. v2 also upgrades sub-requirements to structured Given/When/Then acceptance criteria and introduces cross-sub integration journeys.

## What changed at a glance

| Area | v1 | v2 |
|------|----|----|
| Tasks location | `spec.tasks[]` | Separate `plan.json` (next to `spec.json`) |
| Sub-requirement GWT | Optional | **Required** (`given`, `when`, `then`) |
| Integration scenarios | — | `verification.journeys[]` (optional) |
| Schema marker | Implicit | `meta.schema_version: "v2"` |
| Task CLI | `spec task`, `spec derive-tasks` | `plan init`, `plan get`, `plan status`, `plan list`, `plan merge` |
| Learnings/issues task ref | `spec.tasks[].id` | `plan.json` task IDs |
| Learnings/issues files | Alongside `spec.json` | Alongside `plan.json` (same directory) |

The v1 JSON Schema is preserved at `cli/schemas/_deprecated/dev-spec-v1.schema.json` for reference.

## Step-by-step for existing specs

### 0. Auto-migration (read side)

`hoyeon-cli` performs in-memory auto-migration when loading a v1 spec (see `cli/src/handlers/spec.js:47-75`). Reading a v1 spec with any `spec <subcommand>` will not fail — it is transparently read-upgraded to v2 shape for the duration of the call.

However, **the next `spec merge` writes the file back in v2 shape**. So the first mutation is effectively your migration point. Plan your migration before that first write.

### 1. Add `schema_version` to `meta`

Before:
```json
{
  "meta": {
    "goal": "Add magic-link auth",
    "type": "dev"
  }
}
```

After:
```json
{
  "meta": {
    "goal": "Add magic-link auth",
    "type": "dev",
    "schema_version": "v2"
  }
}
```

### 2. Add GWT to every sub-requirement

v2 requires `given`, `when`, `then` on every sub-requirement. If you previously relied on prose `behavior` alone, split it into the three axes.

Before (v1):
```json
{
  "id": "R1",
  "behavior": "User can request a magic link",
  "sub": [
    { "id": "R1.1", "behavior": "POST /auth/magic-link accepts an email" }
  ]
}
```

After (v2):
```json
{
  "id": "R1",
  "behavior": "User can request a magic link",
  "sub": [
    {
      "id": "R1.1",
      "behavior": "POST /auth/magic-link accepts an email",
      "given": "a registered user email",
      "when": "POST /auth/magic-link is called with that email",
      "then": "a signed token is emailed and the response is 202 Accepted"
    }
  ]
}
```

Use `hoyeon-cli spec guide sub` if you forget the field names.

### 3. Move tasks out of `spec.json` into `plan.json`

If your v1 `spec.tasks[]` contains work breakdowns, move them to a sibling `plan.json`.

```bash
# From the directory containing spec.json
hoyeon-cli plan init .

# Merge tasks in (one call per top-level section)
cat > /tmp/plan-merge.json << 'EOF'
{
  "tasks": [
    {
      "id": "T1",
      "action": "Implement POST /auth/magic-link handler",
      "type": "code",
      "status": "pending",
      "depends_on": [],
      "fulfills": ["R1.1"]
    }
  ]
}
EOF
hoyeon-cli plan merge --json "$(cat /tmp/plan-merge.json)" .
```

Then delete `tasks[]` from `spec.json`. After the next `spec merge`, v2 will refuse to re-accept `tasks` in spec shape anyway.

### 4. (Optional) Add integration journeys

If you have cross-sub scenarios that only make sense when multiple sub-requirements compose, add them under `verification.journeys[]`.

```json
{
  "verification": {
    "journeys": [
      {
        "id": "J1",
        "name": "First-time login end-to-end",
        "composes": ["R1.1", "R1.2", "R2.1"],
        "given": "a user with no prior session",
        "when": "they request a magic link and click it",
        "then": "they land on /dashboard with a valid session cookie"
      }
    ]
  }
}
```

Journeys are optional — omit the field entirely if you don't need them.

## Running `/execute` after migration

In v2, `/execute` takes the spec path as a CLI argument and treats `plan.json` as a generated byproduct that lives next to `spec.json`.

```bash
/execute .hoyeon/specs/my-feature/spec.json
```

If `plan.json` does not exist yet, `/execute` will initialize one. If it exists, `/execute` reads it as the current task breakdown. You can regenerate `plan.json` freely without touching `spec.json`.

Learnings and issues emitted during execution are now written alongside `plan.json` (same directory as `spec.json`):

```bash
hoyeon-cli spec learning --task T1 --stdin .hoyeon/specs/my-feature/spec.json << 'EOF'
{"problem": "...", "cause": "...", "rule": "...", "tags": ["auth"]}
EOF
```

The `--task T1` reference now resolves against `plan.json`, not `spec.tasks[]`.

## FAQ

**Q: Can I keep `tasks[]` inside `spec.json`?**
No. v2 rejects `tasks[]` under `spec.json`. Auto-migration strips it on read, and the next `spec merge` persists the stripped version. Move tasks to `plan.json`.

**Q: What if my sub-requirement legitimately has no GWT (e.g., a pure non-functional constraint)?**
Non-functional constraints belong in `constraints[]`, not `sub[]`. If it really is a sub-requirement, you can almost always articulate it as GWT — even "given the deployed service, when a request is made, then response time is under 200ms" is a valid GWT. If you're stuck, run `hoyeon-cli spec guide sub` for templates.

**Q: How do I verify `verification.journeys[]`?**
`/execute` picks them up during the verify stage. Each journey becomes an integration test target that exercises its `composes[]` sub-requirements end-to-end. They are orthogonal to per-sub unit verification.

**Q: Do I have to migrate all my specs at once?**
No. Auto-migration on read means v1 specs keep working until you next `spec merge` them. Migrate on demand, one spec at a time.

**Q: How do I roll back?**
Restore the file from git and pin usage to an older `hoyeon-cli` version. The v1 schema is preserved at `cli/schemas/_deprecated/dev-spec-v1.schema.json` for reference, but v2 tooling will always read-upgrade v1 input.

**Q: Where did `spec derive-tasks` go?**
It's now `plan init` followed by `plan merge` with your derived tasks. Task derivation logic is unchanged — only the storage location moved.
